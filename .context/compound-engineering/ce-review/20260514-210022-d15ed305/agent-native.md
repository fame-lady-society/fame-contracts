Agent-native review against base 1f6086a8570c33d9a54a43abcbec9f0b17bd389a.

Overall: the branch is much more agent-operable than a prose-only router launch would be. It adds root-level `bun run router:*` commands, deterministic JSON artifacts with route hashes/funding/debug metadata, generated Solidity hash manifests, durable plans/todos, Doppler/public-config conventions, and validation scripts. The remaining gaps are around production replayability and release gates that still require hidden human context.

Findings/gaps:

P1 - Generated Universal Router artifacts are executable only in the test address context, not against the deployed router. `router-ts/src/compiler/compileRoute.ts:25` hard-codes `ROUTER_TEST_EXECUTOR = 0x...f00d`, `DEFAULT_COMPILE_CONTEXT` uses it at `router-ts/src/compiler/compileRoute.ts:34`, and V3/V4 payload generation passes that executor as the Universal Router recipient through `buildLeg`/`buildCompiledRoute` at `router-ts/src/compiler/compileRoute.ts:79` and `router-ts/src/compiler/compileRoute.ts:129`. The onchain adapter rejects payloads whose embedded recipient is not the executing router at `src/router/adapters/UniversalRouterAdapter.sol:72` and `src/router/adapters/UniversalRouterAdapter.sol:102`. The fork test masks this by deploying `FameRouter` to `GENERATED_ROUTER` before replaying artifacts at `test/router/FameRouterForkBase.t.sol:457`. An agent can regenerate and hash-check artifacts, but cannot safely replay the checked-in V3/V4 artifacts against `BASE_FAME_ROUTER_ADDRESS` without knowing this test-only address coupling. Fix: make the generator accept an explicit executor/router address, persist that context in the artifact set, add a `--check --executor "$BASE_FAME_ROUTER_ADDRESS"` style command, and validate generated Universal Router artifacts against the same address used by deployment validation.

P2 - The default documented test command is not a reproducible no-secret command once the manifest is launchable. README labels `forge test` as "Unit tests" at `README.md:52`, but `test_BaseForkLaunchGateRequiresPinnedBlockAndRpc` requires `BASE_RPC` and reverts when missing at `test/router/FameRouterForkBase.t.sol:84`. Other fork tests skip without RPC, so this one gate makes a normal agent run fail unless it has Doppler/Base RPC configured. Fix: split pure/default checks from launch/fork gates in scripts and docs, e.g. `forge test --no-match-path test/router/FameRouterForkBase.t.sol` or a named package script for local verification, plus a separate Doppler-backed launch validation command that intentionally requires `BASE_RPC`.

P2 - Go/no-go still has manual-only release state with no machine-readable completion artifact. The docs require `www` schema parity, deployed `getSkipNFT(router)`, NativeWrap/Aerodrome app enablement checks, and ownership timing at `docs/router/fame-router-validation.md:83`, especially `docs/router/fame-router-validation.md:91` through `docs/router/fame-router-validation.md:95`. Some of these checks are validated by scripts, but cross-repo `www` parity and final ownership readiness are prose bullets only. An agent has no durable status file or command output to mark each gate pass/fail before transfer. Fix: add a checked schema for a go/no-go report and a script or documented command set that emits machine-readable results for contract validation, deployed router address, `www` parity evidence, app enablement blockers, skip-NFT state, and ownership-transfer readiness.

P3 - The validation workflow wording is contradictory around deployment order. `docs/router/fame-router-validation.md:24` says to run live predeployment validation before deployment, but `ValidateFameRouterBase.run()` requires `BASE_FAME_ROUTER_ADDRESS` and checks the deployed router at `script/ValidateFameRouterBase.s.sol:57`, while the command block runs deploy before validate at `docs/router/fame-router-validation.md:79`. This is small, but agents follow literal order. Fix: split "pre-deploy current-pool validation" from "post-deploy router validation", or rename the existing step as post-deploy/live validation.

P3 - The root README deploy path omits the validation command. `README.md:84` through `README.md:93` shows deployment only, while the richer router validation doc includes `ValidateFameRouterBase`. Since README is the first operational surface an agent will try, it should either link directly to the router launch gate from the deploy section or include the deploy-then-validate command sequence.
